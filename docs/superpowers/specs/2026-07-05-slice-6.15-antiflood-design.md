# Slice 6.15 ‚Äî `antiflood`: chat flood protection

**Status:** design (brainstormed 2026-07-05)

**Goal:** Ship SourceMod's `antiflood` as `@s2script/antiflood` ‚Äî a passive chat-flood moderator that throttles a client spamming say/say_team. It is the first **non-command** base plugin (no `Commands.register*`): pure `Chat.onMessage` moderation. This both adds a real SM default plugin and validates that the Slice-6.13-B raw-chat subscriber generalizes beyond the `@` triggers.

**Architecture:** A single `Chat.onMessage` subscriber running a pure token-decay flood model over per-slot state; a flooded message is suppressed by returning `HookResult.Handled`. Config-driven (`@s2script/config`), admin-exempt (`@s2script/admin`). **No new native, no RE, no sniper rebuild** ‚Äî a pure game-package/plugin slice.

---

## Why this plugin now

We just built `Chat.onMessage` (raw-chat subscriber, returns `HookResult` to suppress) for basechat's `@` triggers. `antiflood` exercises it in a completely different shape ‚Äî a passive moderator with no commands ‚Äî which is the cheapest way to prove the hook is general. Low risk, fast, self-contained.

---

## Architecture ‚Äî three units

### 1. The pure flood model (unit-testable, no engine)

A pure function decides, per message, whether to block ‚Äî mirroring the `activity.js` dual-mode + node:test pattern (a `games/cs2/js/`‚Ä¶ no ‚Äî this is engine-generic, so it lives in the plugin OR a small pure helper the plugin + test share).

`floodStep(state, now, floodTime, maxTokens) ‚Üí { block: boolean, state: newState }` where `state = { tokens: number, lastTime: number }`:
- `dt = now - state.lastTime`.
- If `dt < floodTime` ‚Üí `tokens = state.tokens + 1` (too fast ‚Üí accrue credit).
- else ‚Üí `tokens = max(0, state.tokens - 1)` (well-spaced ‚Üí decay one).
- `lastTime = now`.
- `block = tokens >= maxTokens`.
- SM parity: `sm_flood_time = 0` disables entirely (return `{block:false}` unchanged).

Pure ‚Üí fully node:test-covered across {burst, spaced, recovery-after-decay, disabled}. Keep it a small standalone module both the plugin and the test import (dual-mode, like `activity.js`), so it's testable without the engine.

### 2. The time source (exploration ‚Äî pick the self-contained one)

The model needs a monotonic "now". In priority order, use whichever the plugin runtime actually provides (an **implementation spike**, no new primitive either way):
1. **`Date.now()`** ‚Äî standard V8; wall-clock ms is the right granularity for real-time flood spacing. Verify it exists + isn't stripped in the plugin sandbox. If present, use it (`now` in seconds = `Date.now()/1000`).
2. **Fallback: an `OnGameFrame` frame counter** (`@s2script/frame`) ‚Äî increment a module counter each frame; `now = frames / TICKRATE` (64). Self-contained, uses the frame multiplexer we already have. Slightly heavier (a per-frame handler) but no new primitive.

The design does not depend on which ‚Äî the flood model takes `now` as a parameter. Document the chosen source.

### 3. The plugin: `@s2script/antiflood`

- **Config (`@s2script/config`):** `flood_time` (float, default `0.75`) and `max_tokens` (int, default `3`) declared in `package.json`'s `s2script.config`; read at load, honor `config.onChange` for live-reload (opt-in ‚Äî the author decides; include it since flood tuning is exactly what an admin edits live).
- **Subscriber:** `Chat.onMessage((slot, text, teamonly) => ‚Ä¶)`:
  - If `flood_time <= 0` ‚Üí `HookResult.Continue` (disabled).
  - **Admin exemption:** `Admin.forSlot(slot)?.hasFlags(ADMFLAG.CHAT)` ‚Üí `HookResult.Continue` (admins aren't throttled ‚Äî SM-reasonable; keeps admins usable during a flood). *(Revisit: a dedicated immunity flag if one emerges; `ADMFLAG.CHAT`/GENERIC is the pragmatic choice.)*
  - Else run `floodStep` on the per-slot state ‚Üí if `block`, `HookResult.Handled` (suppress the message); else `HookResult.Continue`.
- **Per-slot state:** a `Map`/array keyed by slot. Slot reuse across reconnects is benign (tokens decay in a few spaced messages); optionally reset a slot on `Events.on("player_disconnect")` (cheap, we have the event bus) ‚Äî include it to be tidy.
- **No commands, no chat output** on block (SM silently drops the flooded line; optionally a throttled "you are being throttled" notice ‚Äî deferred to keep it minimal).

## Data flow

`say <text>` ‚Üí `Host_Say` detour ‚Üí `dispatch_chat` (not a `!`/`/` command) ‚Üí `CHAT_MSG_SUBS` ‚Üí antiflood `onMessage` ‚Üí (disabled? admin? ‚Üí Continue) ‚Üí `floodStep(now)` ‚Üí block? ‚Üí `HookResult.Handled` suppresses the broadcast : `Continue` shows it.

## Error handling / degrade

- Missing/garbage config ‚Üí the `@s2script/config` type-zero/default already degrades (5E.2). `flood_time=0` ‚Üí disabled.
- Unknown/absent admin lookup ‚Üí not exempt (flood-checked), never a crash.
- A handler throw ‚Üí the `CHAT_MSG_SUBS` dispatch already `log_warn`s + treats as Continue (message shows) ‚Äî fail-open (never lock out chat on a bug).

## Testing

- **Unit (node:test):** `floodStep` across burst (accrues ‚Üí blocks at maxTokens), spaced (decays, never blocks), recovery (blocked then spaced ‚Üí unblocks), `flood_time=0` (disabled passthrough), boundary (exactly maxTokens).
- **Typecheck gate:** `plugins/antiflood` full-strict.
- **Live gate (de_dust2, a client + the time-source spike resolved):** spam chat rapidly ‚Üí after `max_tokens` fast lines the messages stop appearing (suppressed); slow, spaced messages always show; an admin is never throttled; `RestartCount=0`. No sniper (JS-only) ‚Äî a `.s2sp` drop + restart.

## Out of scope (later)

A "you're being throttled" notice; per-flag immunity tiers; flood protection for commands/voice; `sm_flood_time` as a real registered cvar (config file is the tuning surface here).
