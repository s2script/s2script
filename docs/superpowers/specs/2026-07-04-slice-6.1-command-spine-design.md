# Slice 6.1 — command spine + chat messaging (design)

**Goal:** The first rung of the base-plugin suite (Slice 6): a working **command spine** — register a
server command, dispatch it to a typed JS handler, route a reply — plus **chat messaging**
(`PrintToChat`), proven end-to-end by a real `@s2script/basecommands` plugin exposing **`sm_say`**
(broadcast a message to all players). Ungated this slice (admin auth is 6.2).

**Status:** design approved (spine-first 3-slice decomposition: 6.1 spine+messaging → 6.2 admin cache +
flags → 6.3 kick + slap actions; the two engine-RE pieces discovered during the tasks, not a separate
spike).

**Branch base:** `main` (… + 5E.3 merged).
**Cadence:** subagent-driven, merge-to-main-locally, Docker CS2 live gate.

The base-plugin suite is the std lib's acceptance test (the charter): awkwardness building `sm_say` on the
runtime is a std-lib bug. This slice builds by risk — the smallest real command, end-to-end, before
breadth.

---

## 0. Starting point (what already exists)

- **The concommand seed is assembled but NEUTRALIZED.** `shim/src/s2script_mm.cpp::s2_concommand_register`
  currently logs a WARN and registers nothing — `ConCommand::Create` is not exported by CS2 (a confirmed
  `dlopen` blocker). BUT the machinery around it is intact and correct: a shared `s2_concommand_trampoline`
  reads `(name, slot, args)` (slot = `-1` for the server console) and calls
  `s2script_core_dispatch_concommand(name, slot, args)` → `v8host::dispatch_concommand` → the registered
  JS `Global<Function>`. The `__s2_concommand(name, fn)` native + the `concommand_register` op + the
  `s_concommands` vector all exist. **Only the registration body is stubbed.**
- **No player-facing messaging exists** — `console.log` prints to the *server* console; nothing prints to
  a *client's* chat.

## 1. The two engine-RE pieces (folded into the tasks)

Both are proven feasible (every CS2 framework — CSSharp, Swiftly, Metamod plugins — does them); this slice
finds the CS2-specific path, the same treadmill work as 5D.2's sig-scan. Findings become committed
gamedata (signatures/offsets/interface strings), never hardcoded in C++/Rust.

1. **Make ConCommand registration work.** Replace the neutralized `s2_concommand_register` body with a real
   CS2 registration path — candidates, in order of preference: an exported `ICvar::RegisterConCommand` /
   `CCvar` method (interface-routed, most stable); a `ConCommandRef`/`ConCommandCreation_t` construction; a
   vtable/sig-scanned `ConCommand::Create` equivalent (last resort). The trampoline + dispatch already
   consume the result. **Teardown:** unregister each `s_concommands` entry on plugin unload (the existing
   `TODO(teardown)`), so a reload doesn't leak/duplicate commands.
2. **`PrintToChat`.** A new engine-op + native `__s2_client_print(slot, msg)` (print to one client's chat)
   the shim implements via the CS2 chat path — candidates: the `SayText2`/`TextMsg` user message through
   `IGameEventSystem::PostEventAbstract` (the CSSharp path), or an engine/gamedll client-print. Broadcast
   = the native looped over live slots (a to-all user message is an optimization, deferred).

## 2. Components & boundary

| Concern | Lives in | Engine-generic? |
|---|---|---|
| ConCommand registration path + teardown; chat user-message send | shim (`s2script_mm.cpp`) + committed gamedata | shim owns the CS2 specifics |
| `concommand_register` op (exists) + new `client_print` op (ABI-appended); the `__s2_concommand` (exists) + `__s2_client_print` natives | core (`v8host.rs`) | yes — `(slot, string)`, no game symbol |
| `@s2script/commands` module (`Commands.register` + the dispatch `ctx`); `@s2script/chat` module (`Chat.toSlot`/`toAll`) | core prelude (`__s2pkg_commands`/`__s2pkg_chat`) | yes — a console command + a client-print are Source2 concepts |
| `@s2script/basecommands` plugin (`sm_say`) | game package / registry (`@s2script/base*`) | no — a CS2 plugin |

**No CS2 symbol enters `core/src`** (both boundary gates stay green). The base plugin is a CS2 plugin
(`@s2script/base*`), registry-distributed, not built into the runtime (charter).

## 3. The `@s2script/commands` API

```ts
Commands.register(name: string, handler: (ctx: CommandContext) => void): void;

interface CommandContext {
  callerSlot: number;         // 0-based player slot, or -1 for the server console
  args: string[];             // argString split on whitespace (quotes respected — minimal)
  argString: string;          // everything after the command name (raw)
  reply(message: string): void; // routes to the caller: server console → server print; a player → their chat
}
```

- Engine-generic on purpose: `ctx.callerSlot` is a raw slot (not a `Player`) so `@s2script/commands`
  never depends on `@s2script/cs2`. A CS2 plugin that wants the caller as a `Player` does
  `Player.fromSlot(ctx.callerSlot)` itself (6.3 needs this; 6.1's `sm_say` does not).
- `reply(msg)`: `callerSlot < 0` → server-console print (via `console.log`, which already targets the
  server console); else → `Chat.toSlot(callerSlot, msg)`. (RCON-response routing — SM's `ReplyToCommand`
  to an rcon socket — is a later refinement; a console reply is the first cut.)
- Registration is auto-ledgered (like event subs): a plugin's commands are unregistered on unload
  (Task 1's teardown), so a hot-reload re-registers cleanly.

## 4. The `@s2script/chat` API

```ts
Chat.toSlot(slot: number, message: string): void;   // one client's chat
Chat.toAll(message: string): void;                    // every live player's chat
```

Thin wrappers over `__s2_client_print`. Plain-string messages this slice; CS2 color codes / the `[SM]`
tag are the plugin's concern (it prepends the tag), not the module's. `toAll` iterates slots **in core**
gated by the engine-generic `__s2_client_valid` (NOT `Player.all()` — that is CS2, and `@s2script/chat`
must stay engine-generic / free of any `@s2script/cs2` dependency).

## 5. The `sm_say` demo (`@s2script/basecommands`)

```ts
import { Commands } from "@s2script/commands";
import { Chat } from "@s2script/chat";

export function onLoad(): void {
  Commands.register("sm_say", (ctx) => {
    const msg = ctx.argString.trim();
    if (!msg) { ctx.reply("Usage: sm_say <message>"); return; }
    Chat.toAll("[SM] " + msg);
  });
}
```

Ungated (6.2 adds `ADMFLAG_CHAT`). Chat triggers (`!say` in the chat box, via hooking `say`) are **out of
scope** for 6.1 — `sm_say` is invoked as a server/console command; the say-hook is a separate concern
(6.1b or folded into 6.2/6.3). This keeps 6.1 to the two RE pieces above.

## 6. Degrade-never-crash

Registration path unavailable (RE not yet wired / a bad build) → `__s2_concommand` WARNs, the command is
absent, the plugin still loads (today's behaviour, preserved). `__s2_client_print` with no ops / an
invalid slot → no-op, no crash. An empty/garbage arg → the handler's own guard (`Usage:` reply). A
throwing handler → caught in `dispatch_concommand` (mirror the event-dispatch TryCatch), WARN, server
ticks on.

## 7. Live gate & command delivery

- **Command delivery (a gate prerequisite):** invoking `sm_say` on the live server needs either RCON (a
  small client — the server has `CS2_RCONPW`) or the server-console stdin (`docker attach` / writing the
  container's PID-1 stdin). Task 4 solves this (a reusable `scripts/rcon` one-liner is the clean option;
  it also unblocks future gates that need `bot_quota`/`bot_kick`, which 5E.3 worked around).
- **Live gate (de_inferno):** load `@s2script/basecommands`; run `sm_say hello world` (server
  console/rcon) → the JS handler fires (`[basecommands] sm_say …` log), `Chat.toAll` sends the user
  message, server ticking, no crash. **Visual confirmation caveat:** bots have no chat UI to read, so
  full "a player saw it" verification needs a connected human client (best-effort); the gate's hard
  assertions are: registration succeeded (the command dispatched — proving RE piece 1), the print native
  was invoked without error (proving RE piece 2 wired), and no crash. If a human client is available,
  confirm the chat line visually.
- **Reload:** re-run the plugin drop → the command re-registers (teardown + re-register), no duplicate.

## 8. Testing

- **In-isolate (core):** `__s2_concommand` registers → a simulated `dispatch_concommand(name, slot, args)`
  invokes the handler with the right `ctx` (callerSlot -1 vs a slot; args split; argString); `reply`
  routes (console vs `__s2_client_print`); a throwing handler is caught; unregister-on-unload clears the
  handler (a post-unload dispatch is a no-op). `Commands`/`Chat` prelude module shapes.
- **vm (CLI/JS):** the `@s2script/commands` + `@s2script/chat` `.d.ts` compile; `basecommands` passes the
  5E.1 typecheck gate.
- **One sniper rebuild** (the registration path + the `client_print` op/native — shim + core).

## 9. Rough task decomposition (~4)

1. **ConCommand registration (RE + wire) + teardown.** Replace the neutralized `s2_concommand_register`
   with the CS2 path (committed gamedata sig/interface); unregister `s_concommands` on unload. Gate:
   cargo (the native path) + a sniper build proves live registration + dispatch of a bare test command.
2. **`PrintToChat` native.** `client_print` op (ABI-appended, C header + Rust mirror) + shim
   chat-user-message impl (committed gamedata) + the `__s2_client_print` core native. Gate: cargo; live
   print.
3. **`@s2script/commands` + `@s2script/chat` prelude modules + `.d.ts` + reply routing + the
   `dispatch_concommand` `ctx` + TryCatch + unregister-on-unload wiring.** In-isolate + vm tests.
4. **`@s2script/basecommands` (`sm_say`) + `scripts/rcon` command delivery + one sniper build + live gate +
   docs.** Controller-driven.
